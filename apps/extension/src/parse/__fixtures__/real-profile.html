    AI can write the code. It can't tell you the code shouldn't exist.

    It'll scaffold services, generate schemas, and wire up integrations — all to spec. What it won't do is notice when the spec itself is wrong.

    That's where the mess comes from. Not model limitations, but accidental complexity that scales: dependency trees nobody can explain, abstractions that justify nothing, three services doing one service's job.

    AI is a force multiplier. Point it at a well-defined problem, and the results are remarkable. Point it at a vague one, and it gives you something worse than wrong — something plausible. Solutions that pass the tests and quietly become tech debt.

    The bottleneck was never implementation. It's modelling: deciding what to build, what to throw away, and where the boundaries sit. That part is still yours.

    So the question stops being "Can AI build this?" and goes back to the one engineering always asked: Is it actually sound?

    Stop asking when. Start asking why.

    What's the most "plausible but wrong" thing you've watched AI confidently produce?

    One week. Claude Code. Claude Opus 4.7. One question: what can you actually ship? That's Built with Opus 4.7 — a hackathon by Anthropic and Cerebral Valley. Here's mine.

    I'm building Vastify – CRM storage and backup, with Claude inside.

    It's a transparent middleware that sits between Salesforce and a customer-owned cloud bucket (S3, GCS, Azure, R2, MinIO). A hand-written OData 4.0 adapter speaks the same wire protocol Salesforce expects from External Objects, so the consuming apps can't tell the data has left the org — but the bytes now live in AES-256-encrypted, tier-routed object storage at roughly 1/100th the cost of native Salesforce storage.

    The interesting part for the hackathon is three Claude Opus 4.7 agents wired up through the Anthropic Agent SDK:

    1. A Setup Agent with 6 typed tools that installs the whole platform end-to-end in 42 seconds — inspects the org schema, picks the cheapest backend for the data shape it finds, writes encrypted credentials sealed with the tenant's KMS key, generates starter routing rules, runs sf project deploy start, and validates the OData endpoint. Each tool call streams to the dashboard as a live transcript over SSE.

    2. A Diff Explainer that reads a backup change-set + recent commit history and returns a structured DiffExplanation with per-object verdicts and reasoning. Across hundreds of test runs the structured outputs have been near-100% reliable.

    3. A Rule Generator that turns natural language ("move all contracts older than 90 days to Glacier and ping the AE in Slack") into JSON-schema-validated routing rules, with automatic retry on validation failure.

    A few honest observations on building with Opus 4.7 via Claude Code:

    → Long, vague design conversations actually converged on real decisions instead of devolving into wishy-washy options. The OData ↔ SQLite translator design got hashed out in about 40 minutes of back-and-forth.

    → The agent loop recovered gracefully from intermediate tool errors — when deploy_sf_package returned a transient INVALID_SESSION_ID, the agent re-authenticated and continued without prompting.

    → Cross-file consistency between API types, UI types, and Apex metadata is where it really shines. That's the difference between demo-working and demo-working-AND-maintainable.
